# AFSIM_MCP Capability Assessment

本文档用于回答三个问题：

1. 这个 MCP 目前真实能做到什么程度。
2. 它的工作效果与接入的大模型关系有多大。
3. 它是否已经达到“用户用自然语言发起 AFSIM 项目”这一目标。

## Overall Judgment

当前 AFSIM_MCP 已经具备一条较完整的工作闭环：

- 粗提示词整理与补全
- 任务计划生成
- 场景文件与项目结构生成
- briefing / phases / prompt brief / showcase 文档生成
- mission / wizard / mystic / warlock / engage 运行控制
- EVT / AER / SENSOR 输出分析

这意味着它已经不是“只会吐模板文本”的原型，而是一个中等成熟度的 AFSIM 工程辅助 MCP。

但它还不是一个“只给一句模糊需求就能稳定产出高真实性完整工程”的全自动系统。它更接近：

- 一个有 prompt refinement 和 task planning 的 AFSIM 工程助理
- 一个能把粗需求整理成可运行、可讲解、可回放项目骨架的工具链
- 一个对接好模型后能明显提升项目交付效率的工程接口

## Requirement Fit

### 1. 用户只说一句粗需求时

目标示例：

我要做一个 AFSIM 项目，空军大战。

当前状态：部分达到，接近可用。

已具备的能力：

- 能对输入做类型分类，例如更偏制空空战、空袭打击或防空反导。
- 能推断默认时长、中心区域、replay focus、KPI。
- 能给出 `low_confidence` 和 `questions_needed`。
- 能把粗输入扩展为 `recommended_model`，再进入项目生成。

当前不足：

- 是否先追问用户，不由 MCP 强制决定，而取决于接入模型是否正确调用 refinement 工具。
- 模糊输入下的战区推断仍然偏启发式，默认中心点可能不符合用户真实意图。
- 当前 `suggest_scenario_questions` 偏通用表单；真正更强的是 `refine_operational_prompt`，但这要求模型先走对流程。

结论：

- 如果接入模型会先 refinement 再追问，这个场景效果是好的。
- 如果接入模型直接跳过 refinement 并立刻生成，效果会明显下降。

评级：

- 工作流能力：已达到
- 稳定用户体验：部分达到
- 开箱即用程度：部分达到

### 2. 用户确认方向后，进一步明确任务、环境、平台、组件、剧本

目标示例：

- 明确任务场景环境
- 选平台类型
- 定义 mover / sensor / weapon / processor 等数据
- 当用户给出具体平台和组件数据后，把这些信息映射到 AFSIM 项目

当前状态：作战层面已达到中等偏上可用，组件参数工程化层面部分达到。

已具备的能力：

- 能从 prompt 或 operational_model 构造 task plan。
- 能生成 force packages、phases、engagement rules、target priority、route style、timing notes。
- 能生成 entry 文件、scenario 文件、briefing、phases、prompt brief、showcase。
- 能提供平台、传感器、武器、mover 的模板生成工具。
- 能搜索项目中的定义块，帮助复用已有资产。

当前不足：

- 没有内置高可信的武器/传感器/平台参数知识库。
- 不会自动验证用户给出的组件参数是否满足工程一致性、真实性或战术合理性。
- 对“完整组件级参数设计”的支持更像模板拼装和结构辅助，而不是严格的专业建模引擎。

结论：

- 如果用户给的是任务级需求，它已经能较好生成项目骨架和剧本。
- 如果用户给的是组件级详细数据，它可以辅助落盘和组织结构，但还需要模型具备较强的 AFSIM 建模理解能力。

评级：

- 作战任务建模：已达到
- 项目骨架生成：已达到
- 组件模板辅助：已达到
- 高真实性参数建模：部分达到
- 自动工程校核：未达到

## Dependency On The Connected LLM

这个 MCP 的工作效果与接入的大模型关系很大，而且是关键性关系。

### 与模型高度相关的部分

1. 是否先走正确流程

- 好模型会先调用 `refine_operational_prompt`
- 看到 `low_confidence` 后会继续追问
- 用户确认后才调用 `create_operational_scenario_package`

2. 是否理解返回结果

- MCP 会返回 `questions_needed`、`filled_by_system`、`confidence`
- 模型需要把这些结构化结果转成用户能理解的追问或建议

3. 是否能处理详细工程信息

- 当用户给平台、传感器、武器、处理器等细节时
- 模型需要把这些信息拆解成适合调用模板工具、搜索定义工具、插入场景块工具的步骤

### 与模型弱相关的部分

以下能力主要由 MCP 自身提供：

- 项目结构创建
- 场景文本落盘
- briefing / phases / prompt brief / showcase 生成
- 仿真运行与进程管理
- 输出分析

### 实际判断

如果接入模型较强，并且系统提示要求它遵循标准工作流，整体效果会明显提升一个档次。

如果接入模型较弱，或者没有系统提示要求它先 refinement 再追问，那么这个 MCP 的上限虽然还在，但用户最终看到的效果会显著打折。

## Status Matrix

### 已达到

- 粗提示词转结构化任务模型
- 任务计划转项目骨架
- 生成可讲解的 briefing / phases / showcase 文档
- 运行 mission 并分析输出
- 支持回放与 KPI 导向分析

### 部分达到

- 模糊输入下的稳定追问体验
- 战区与资产选择的高置信度推断
- 用户给定组件数据后的自动工程化落盘
- realism 与演示效果兼顾时的自动权衡

### 未达到

- 不追问也能稳定生成高质量工程级完整项目
- 自动完成高真实性平台/传感器/武器参数设计
- 自动完成严格工程一致性校核

## Final Answer

如果你的目标是：

- 用户与大模型说一句较粗的话
- 大模型先整理意图、适度追问、给出更好的 prompt 或直接形成推荐模型
- 再生成一版可运行、可讲解、可回放的 AFSIM 项目

那么当前项目已经基本达到“可以投入使用并继续迭代优化”的阶段。

如果你的目标是：

- 用户只给一句非常模糊的话
- 系统几乎不追问
- 自动完成高真实性、工程级、组件参数完整的 AFSIM 项目

那么当前项目还没有完全达到。

最准确的表述是：

- 它已经具备一条靠谱的 AFSIM 项目生成与分析闭环。
- 它的最终效果高度依赖接入模型的质量与工作流约束。
- 它已经适合做“可运行 + 可讲解 + 可回放”的项目型交付。
- 它还不适合被当成“无需追问即可自动完成全部高真实性建模”的黑盒系统。
